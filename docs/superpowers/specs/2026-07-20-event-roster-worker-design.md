# Event Roster Worker MVP 설계

- 작성일: 2026-07-20
- 상태: 사용자 승인 완료
- 대체 대상: `2026-07-20-event-roster-design.md`의 FastAPI Python Worker 설계 및 그 구현 계획
- 대상: 상반기·하반기에 각각 한 번 운영하는 내부 행사 참가자 명단 서비스

## 1. 목표와 범위

이 서비스는 행사 전 조직별 예상 참가자 명단과 행사 당일의 추가·취소·수정을 한 곳에서 관리한다. 운영자는 조직별·전체 예상 인원, 당일 변동, 최종 유효 인원을 즉시 확인하고 행사별 명단을 엑셀로 내보낼 수 있어야 한다.

MVP에는 다음을 포함한다.

- 행사 생성, 상태 전환, 마감·재오픈
- 조직별 사전 참가자 등록
- 행사 당일 참여자 추가·취소·수정
- 조직별·전체 예상 및 최종 인원 조회
- 반복 행사에서 재사용하는 참가자 마스터
- 조직별 권한, 변경 감사 이력, 동시 수정 충돌 방지
- 기존 엑셀의 브라우저 기반 이관과 행사별 엑셀 내보내기
- 운영자 생성 계정, 최초 비밀번호 변경, 자체 로그인과 JWT 세션

MVP에서 제외한다.

- 현장 체크인과 실제 출석 수 집계
- 원본 엑셀 파일 보관
- 공개 회원가입, 셀프 비밀번호 찾기, 서비스의 이메일 발송
- 이메일 OTP, Cloudflare Access, 외부 IdP
- 엑셀 파일과 서비스 데이터의 양방향 동기화
- 실시간 공동 편집, 채팅, 알림 자동 발송
- 행사 종료 뒤의 삭제 또는 재사용

`실제 인원`은 체크인 수가 아니라 당일 변경이 반영된 최종 유효 명단 수를 뜻한다.

## 2. 핵심 결정

| 항목 | 결정 |
| --- | --- |
| 행사 주기 | 상반기·하반기 각 1회 |
| 명단 소유 | 모든 명단은 특정 행사에 귀속 |
| 참가자 식별 | 시스템이 발급한 참가자 번호 + 이름 + 현재 소속 조직 |
| 참가자 재사용 | 행사와 독립된 참가자 마스터에서 검색·선택 |
| 수정 권한 | 운영자는 전체, 조직 담당자는 자기 조직만 |
| 변경 이력 | 작업자·시각·변경 유형·전후 값까지 영구 보관 |
| 실제 인원 | 당일 변경이 반영된 최종 유효 명단 수 |
| 예상 인원 | 당일 운영 전환 순간의 조직별 스냅샷 |
| 엑셀 | 원본 파일 미보관, 브라우저에서 변환·검토, 서비스는 데이터만 저장 |
| 인증 | 운영자 생성 임시 비밀번호, 최초 변경 강제, 자체 JWT 세션 |
| 배포 | React/Vite 정적 자산 + Hono TypeScript Worker + D1, 하나의 `workers.dev` 배포 |

## 3. 아키텍처와 모노레포

코드는 프런트엔드와 백엔드를 분리하지만, 배포는 하나의 Cloudflare Worker로 한다.

```text
브라우저
  │ https://event-roster.<account>.workers.dev
  ├── React/Vite 정적 자산
  └── /api/v1/* → Hono TypeScript Worker → D1
```

`workers.dev`는 커스텀 도메인 비용 없이 HTTPS를 제공한다. 이 서비스는 반기 1~2회, 운영 인력 약 13명, 행사당 70~130명 규모의 내부 도구이므로 Workers Free 한도 안에서 운영하는 것을 목표로 한다. SLA가 필요한 핵심 시스템으로 성장하면 커스텀 도메인과 유료 플랜을 별도로 검토한다.

```text
event-roster/
├── apps/
│   ├── web/                 # React/Vite 화면과 브라우저 엑셀 처리
│   └── worker/              # Hono API, 인증, D1 바인딩, 정적 자산 배포 설정
├── packages/
│   ├── domain/              # 순수 행사·명단·권한·집계 규칙
│   └── contracts/           # Zod 스키마와 공유 API 타입
├── docs/
└── package.json             # pnpm 모노레포 루트
```

`apps/web`의 빌드 결과는 Worker Static Assets로 배포한다. `/api/v1/*`만 Worker 라우터가 처리하고, 나머지는 정적 자산으로 응답한다. 화면과 API가 같은 origin을 공유하므로 CORS나 별도 API 도메인이 필요 없다.

라우터는 HTTP 변환·인증 진입·응답 형식만 담당한다. `packages/domain`의 유스케이스는 권한, 행사 상태, 낙관적 잠금, 트랜잭션 경계를 확인하고, D1 호출은 Worker의 인프라 모듈에 둔다. 단일 D1 구현만을 위해 불필요한 저장소 인터페이스 계층은 만들지 않는다.

## 4. 사용자, 인증, 권한

### 역할

#### 운영자

- 행사·조직·사용자 계정을 생성하고 수정한다.
- 모든 조직과 모든 행사 명단을 조회·수정한다.
- 행사 상태를 전환하고, 마감된 행사를 재오픈한다.
- 엑셀 이관·내보내기와 감사 이력 조회를 수행한다.
- 조직 담당자의 임시 비밀번호를 발급하거나 재설정한다.

#### 조직 담당자

- 연결된 자기 조직의 참가자 마스터를 검색하고 새 참가자를 등록한다.
- 자기 조직의 열려 있는 행사 명단만 추가·취소·수정한다.
- 자기 조직의 조직별 집계와 변경 이력을 조회한다.
- 다른 조직의 참가자·명단·감사 이력은 조회하거나 수정할 수 없다.
- 참가자의 이름은 정정할 수 있지만, 참가자 마스터의 현재 소속을 다른 조직으로 옮길 수는 없다. 조직 이동은 운영자만 수행한다.

### 계정 수명주기

공개 가입과 서비스 자체의 이메일 발송은 없다. 운영자가 계정을 만들거나 재설정하면 서버가 CSPRNG로 생성한 20자 이상 임시 비밀번호를 한 번만 표시한다. 운영자는 기존의 안전한 내부 전달 수단으로 그 값을 사용자에게 전달한다. 표시를 닫은 뒤에는 임시 비밀번호를 다시 조회할 수 없으며, 필요하면 새 임시 비밀번호를 발급한다.

임시 비밀번호로 로그인한 사용자는 10분 만료의 `MUST_CHANGE_PASSWORD` 제한 세션만 받는다. 이 세션은 비밀번호 변경과 로그아웃 외의 API를 호출할 수 없다. 새 비밀번호는 Unicode NFC 정규화 뒤 12~128자로 검증한다. 새 비밀번호가 저장되면 서버는 비밀번호 자격 증명을 갱신하고 사용자의 모든 세션을 폐기한 뒤 쿠키를 삭제한다. 사용자는 새 비밀번호로 다시 로그인해야 한다.

정상 로그인 뒤에는 8시간 절대 만료의 서명된 JWT를 `__Host-er_session` 쿠키로만 발급한다. 이 쿠키는 `HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/`를 가지며 `Domain` 속성을 갖지 않는다. JWT에는 사용자 ID, 세션 ID, 세션 버전을 넣고 브라우저 저장소에는 보관하지 않는다. Worker는 서명을 확인한 뒤 `auth_sessions`와 사용자의 세션 버전·활성 상태를 확인한다. 이 D1 확인이 있어야 로그아웃, 비밀번호 재설정, 계정 비활성화, 권한 변경을 즉시 반영할 수 있다.

상태를 바꾸는 요청은 정확히 일치하는 `Origin` 확인과 세션별 CSRF 값의 `X-ER-CSRF` 헤더 검증을 통과해야 한다. 인증된 브라우저는 `GET /api/v1/auth/csrf`에서 새 CSRF 원문을 받아 메모리에만 보관하고, Worker는 그 해시만 세션 레코드에 보관한다. 로그인 실패는 계정 존재 여부를 드러내지 않는 같은 메시지로 응답하고, 존재하지 않는 이메일에도 같은 KDF 검증 경로를 실행한다. 계정은 연속 5회 실패하면 15분 동안 잠기며, 하나의 IP 해시에서는 15분 동안 최대 20회만 시도할 수 있다.

비밀번호 원문, JWT 원문, 비밀번호 해시, salt, pepper는 감사 로그·애플리케이션 로그·엑셀에 절대 포함하지 않는다.

### 비밀번호 기술 검증 게이트

JWT는 로그인 후 세션 표현 방식일 뿐 비밀번호 보안을 해결하지 않는다. 비밀번호에는 빠른 해시를 절대 사용하지 않는다. 사용자별 128비트 CSPRNG salt, Worker Secret에만 있는 pepper, 알고리즘·비용 인자를 함께 저장하는 보안 KDF를 사용한다.

Cloudflare Workers의 native Web Crypto가 지원하는 PBKDF2-HMAC-SHA-256 600,000회가 현재 후보지만, 이는 Workers Free에서 실제로 통과하기 전까지 채택된 것으로 보지 않는다. Argon2id 또는 scrypt WASM을 검토한다면 번들 크기·메모리·CPU를 같은 기준으로 검증한다. KDF 비용을 낮추거나 SHA-256 같은 빠른 해시로 바꿔 Free 한도에 맞추는 것은 금지한다.

보안 KDF가 Free Worker에서 안정적으로 통과하지 않으면 자체 비밀번호 인증을 구현하지 않는다. 그 시점에만 Cloudflare Access/외부 IdP 또는 Workers 유료 플랜을 사용자와 다시 결정한다.

## 5. 행사 수명주기와 명단 규칙

행사는 독립된 최상위 관리 단위다. 기본 제목은 예를 들어 `2026 상반기 행사`이며, 운영자가 제목·행사일·선택 장소를 입력한다. 같은 연도·반기에는 하나의 행사만 만들 수 있다. 기본 목록은 초안·사전 등록·당일 운영 행사만 보이고, 마감 행사는 `지난 행사`에서 읽기 전용으로 조회한다.

```text
초안 → 사전 등록 → 당일 운영 → 마감
                         ↑         │
                         └─ 재오픈 ┘
```

| 상태 | 변경 권한과 동작 |
| --- | --- |
| 초안 | 운영자만 행사 기본 정보를 설정한다. 명단은 아직 편집할 수 없다. |
| 사전 등록 | 운영자와 조직 담당자가 권한 범위에서 명단을 갱신한다. 예상 인원은 현재 유효한 사전 명단 수다. |
| 당일 운영 | 운영자가 전환하는 즉시 조직별 예상 인원 스냅샷을 생성한다. 이후 추가·취소·수정은 변동과 최종 인원에 반영한다. |
| 마감 | 읽기 전용이다. 집계·감사 이력·엑셀 내보내기만 허용한다. |
| 재오픈 | 운영자만 `당일 운영` 상태로 되돌릴 수 있다. 기존 예상 인원 스냅샷은 유지하며 재오픈 사실을 감사 이력에 남긴다. |

사전 등록으로 되돌리는 전이는 제공하지 않는다. 이렇게 해야 당일 비교 기준이 바뀌지 않는다.

각 행사는 참가자당 명단 행을 하나만 가진다. 취소는 삭제가 아니라 `CANCELLED` 상태로 보존한다. 취소 뒤 재참여는 새 행을 만들지 않고 기존 행을 다시 `ACTIVE`로 만든다. 명단 변경은 요청의 `revision`과 최신 값을 비교하며, 다르면 `409 Conflict`와 최신 행·수정자·수정 시각을 반환한다.

당일 운영 전환 순간의 조직별 유효 사전 명단 수가 예상 인원이다. 당일·마감의 최종 인원은 모든 `ACTIVE` 행사 명단 행 수다. 조직별 변동은 예상 기준값과 최종 인원의 차이로 표시한다.

## 6. 데이터 모델

### 사용자와 보안

- `organizations`: 조직 이름, 활성 여부
- `users`: 정규화된 고유 이메일, 표시 이름, 역할(`OPERATOR`, `ORGANIZATION_MANAGER`), 활성 여부, 세션 버전, 생성·수정 시각
- `user_organizations`: 조직 담당자와 조직의 연결. 한 담당자가 여러 조직을 맡을 수 있다.
- `password_credentials`: 사용자별 salt, hash, pepper/KDF 버전·비용 인자, 비밀번호 변경 필요 여부, 변경 시각
- `auth_sessions`: JWT 세션 ID, 사용자, CSRF 값 해시, 발급·만료·폐기 시각
- `login_attempts`: 정규화된 이메일·pepper로 해시한 IP 기준 실패 수와 잠금 만료 시각. 비밀번호나 JWT는 저장하지 않는다.
- `security_events`: 로그인 성공·실패, 계정 잠금, 세션 폐기, 권한 거부를 기록하는 append-only 보안 이벤트. 사용자 ID는 알 수 있을 때만 기록하고 비밀번호·JWT·원본 IP는 저장하지 않는다.

### 참가자와 행사

- `participants`: 시스템 참가자 번호, 이름, 현재 조직, 활성 여부, 생성·수정 시각
- `events`: 제목, 연도, 반기(`H1`, `H2`), 행사일, 선택 장소, 상태, 상태 전환 시각. `(연도, 반기)`는 고유하다.
- `event_roster_entries`: 행사와 참가자를 연결하는 명단 행
  - 참가자 번호 참조
  - 행사 당시 이름과 조직 ID·조직 이름 스냅샷
  - 등록 출처(`PRE_EVENT`, `DAY_OF`)
  - 상태(`ACTIVE`, `CANCELLED`)
  - 낙관적 잠금용 `revision`
  - 생성·수정 시각과 수정 사용자
- `event_expected_snapshots`: 당일 운영 전환 시점의 행사·조직별 예상 유효 인원과 캡처 시각

조직 이동 또는 참가자 이름 수정은 참가자 마스터의 현재 값만 바꾼다. 마감된 행사 명단은 당시 이름·조직 스냅샷을 계속 보인다. 현재 행사에서 기본 정보를 고칠 때는 마스터와 현재 행사 명단 스냅샷을 함께 수정하되, 과거 행사에는 영향을 주지 않는다.

### 감사와 이관

- `audit_logs`: 작업자, 행사, 명단 행 또는 대상 엔터티, 변경 유형, 전 값, 후 값, 발생 시각. 수정·삭제 API가 없는 append-only 데이터다.
- `import_runs`: 행사, 실행 사용자, 처리 행 수, 생성·갱신·오류 수, 확정 시각. 원본 파일·원본 셀 값은 저장하지 않는다.

명단 변경과 그 감사 이력, 이관 확정의 참가자·명단·감사·이관 기록은 모두 하나의 D1 원자 작업으로 저장한다. 이 원자성이 실제 D1에서 실패 시 함께 롤백되는지는 capability gate에서 고의 실패 사례로 검증한다.

## 7. 화면과 디자인 시스템

기본 패턴은 **하이브리드 운영 콘솔**이다. 행사당 70~130명, 동시 운영 인력 약 13명 규모에 맞춰 명단은 카드가 아닌 정보 밀도가 높은 테이블로 제공한다. 이 규모에서는 페이지네이션과 가상 스크롤을 만들지 않는다.

### 주요 화면

- 로그인과 최초 비밀번호 변경
- 행사 목록, 생성, 상태 전환
- 행사 운영 콘솔: 행사 정보·상태, 전체/조직별 예상·최종·변동 요약, 필터 가능한 명단 테이블, 빠른 수정 패널, 최근 변경 이력
- 조직·사용자 관리: 운영자 전용. 조직 연결, 임시 비밀번호 발급, 계정 비활성화
- 엑셀 가져오기: 파일·시트 선택 → 열 매핑 → 오류 해결 → 전체 확정의 집중형 4단계
- 엑셀 내보내기: 선택 행사 기준의 `명단`, `집계` 시트 생성

`coursemos-supporter/docs/design-system.md`의 원칙을 제품에 맞게 이식한다.

- 색상, 간격, 라운드, 그림자, 글자 굵기는 `--er-*` CSS 토큰으로 관리한다.
- `Button`, `Badge`, `Dialog`, `Tabs`, `Table`, `EmptyState`, `Progress`를 React 공통 프리미티브로 만든다.
- 도메인 화면은 프리미티브를 조합하고, 시각 표현은 토큰과 CSS가 담당한다.
- 로딩, 빈 상태, 검증 오류, 권한 없음, 충돌, 처리 완료 상태마다 다음 행동을 명확히 제시한다.

참조 프로젝트의 사이드패널·카드 레이아웃을 그대로 복사하지 않는다. 카드와 단계형 진행은 대시보드·가져오기에 사용하고, 명단 주 화면은 테이블을 쓴다.

## 8. 엑셀 가져오기와 내보내기

엑셀 원본은 브라우저에서만 읽고 서비스에는 보관하지 않는다.

가져오기 흐름은 다음과 같다.

1. 브라우저에서 `.xlsx` 파일과 시트를 선택한다.
2. 이름·조직 열을 매핑하고 샘플 행을 미리 본다.
3. 빈 필수값, 알 수 없는 조직, 같은 이름·조직의 중복, 마스터 후보가 여러 개인 행을 해결한다.
4. 오류가 0개일 때만 정규화된 행을 API에 한 번에 확정한다.

브라우저는 오류와 중복 후보를 먼저 보여 주지만, API도 확정 시 권한·행사 상태·조직 범위·필수값·중복·낙관적 잠금을 다시 검증한다. 오류 행을 건너뛰거나 부분 저장하지 않는다.

내보내기는 브라우저에서 `.xlsx` 파일을 생성한다.

- `명단` 시트: 참가자 번호, 이름, 행사 당시 조직, 등록 출처, 상태, 최종 수정 시각
- `집계` 시트: 조직별 사전 예상, 당일 추가, 당일 취소, 최종 유효 인원

원본 업로드 파일이나 다른 행사의 데이터는 포함하지 않는다.

## 9. 오류 처리와 보안 규칙

| 상황 | 처리 |
| --- | --- |
| 로그인 실패 | 계정 존재 여부를 노출하지 않는 동일한 오류 메시지와 실패 제한 적용 |
| 최초 비밀번호 미변경 | 비밀번호 변경·로그아웃 외 API를 거부 |
| 권한 없는 조직 접근 | `403`으로 거부하고 보안 감사 이벤트를 남김 |
| 마감 행사 수정 | `409`으로 거부하고 마감 상태를 반환 |
| 오래된 명단 저장 | `409`과 최신 행·변경자·수정 시각을 반환하고, 화면에서 병합 또는 새로고침을 안내 |
| 엑셀 검증 오류 | 행 번호·원인·수정 방법을 반환하고 저장하지 않음 |
| 이관 중 서버 실패 | 전체 작업을 실패 처리하고 부분 명단·부분 감사 이력을 남기지 않음 |
| D1·Worker 오류 | 사용자에게 재시도 가능한 일반 메시지, 내부에는 요청 상관관계와 원인 기록 |

## 10. capability gate, 테스트, 운영

### 구현 전 capability gate

기능 구현보다 먼저 최소 TypeScript Worker를 실제 Workers Free `workers.dev`에 배포해 다음을 모두 검증한다.

1. 압축 Worker 번들이 3MB 제한 안에 든다.
2. 후보 KDF가 보안 기준을 지킨 정상 비밀번호, 틀린 비밀번호, 존재하지 않는 계정의 전체 로그인 흐름을 각각 최소 50회 반복해도 CPU 초과 오류를 내지 않고, 전체 로그인 흐름의 p95 CPU 시간이 6ms 이하이다.
3. JWT 서명·검증, D1 세션 폐기, 최초 비밀번호 변경 뒤 모든 기존 세션 거부가 동작한다.
4. 130행 명단 이관과 감사 이력이 D1에서 원자적으로 저장되고, 고의 오류 때 모두 롤백된다.
5. 성공·실패·동시 요청에서 Worker 로그와 지표에 CPU 초과가 없음을 확인한다.

이 중 하나라도 실패하면 이후 기능 구현을 시작하지 않는다. 비용이나 보안을 숨긴 우회가 아니라 인증/배포 선택을 다시 결정한다.

### 테스트 전략

- 도메인·유스케이스: 역할·조직 범위, 행사 상태 전이, 예상 스냅샷, 최종 인원, 취소 보존, 과거 스냅샷, 낙관적 잠금, 원자성
- Worker+D1 통합: 쿠키·JWT·세션 폐기·CSRF·권한 거부·비밀번호 변경·이관 전체 롤백
- React 화면: 로그인 제한 흐름, 테이블 필터, 충돌 안내, 상태별 화면, 가져오기 검증
- 브라우저 E2E: 운영자와 조직 담당자 권한, 행사 전환, 당일 변경, 엑셀 가져오기·내보내기
- 실제 Free Worker: capability gate 및 배포 뒤 핵심 API smoke test

### 운영 절차

VM이 없으므로 OS 패치·서버 백업·인스턴스 모니터링은 하지 않는다. 대신 대량 가져오기 전과 행사 마감 직후 운영자가 서비스의 엑셀 내보내기 파일을 조직의 승인된 보관 위치에 저장한다. 이는 원본 엑셀을 서비스가 보관하는 것이 아니라, 데이터 오입력이나 제한된 DB 복구 기간을 넘는 사고에 대비한 운영 복구 절차다.

서비스 의존성의 보안 업데이트와 Cloudflare 한도 변경은 배포 전 또는 행사 준비 시 확인한다. 사용자 데이터와 비밀값은 Worker 로그에 출력하지 않는다.

## 11. 완료 기준

- `workers.dev` HTTPS 주소에서 운영자와 조직 담당자가 자체 계정으로 로그인한다.
- 최초 로그인 뒤 비밀번호 변경과 강제 로그아웃이 동작한다.
- 운영자는 행사·조직·사용자를 관리하고, 조직 담당자는 자기 조직의 열린 행사만 수정한다.
- 사전 명단, 당일 변동, 예상·최종 인원, 감사 이력, 마감·재오픈 규칙이 일관되게 동작한다.
- 130명 규모의 엑셀 이관·내보내기에서 원본 파일은 서비스에 저장되지 않는다.
- 실제 Workers Free capability gate와 자동화 테스트가 통과한다.
