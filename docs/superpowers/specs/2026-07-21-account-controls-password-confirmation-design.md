# 관리 화면 컨트롤 및 비밀번호 확인 설계

## 목표

관리 화면의 기본 브라우저 입력·선택·체크 컨트롤을 프로젝트 디자인 토큰에 맞춰 일관되게 보이게 한다. 첫 로그인 비밀번호 변경과 복구 코드 재설정에서는 새 비밀번호 확인을 요구해 오입력을 막는다.

## 범위

- 계정 목록의 표시 이름 인라인 입력, 역할 선택, 활성 상태, 담당 조직 선택
- 계정 생성 폼의 역할 선택과 담당 조직 선택
- 조직 관리의 조직 이름 인라인 입력
- 첫 로그인 비밀번호 변경과 복구 코드 비밀번호 재설정의 새 비밀번호 확인

행사·참가자·가져오기 화면의 선택 상자는 이번 변경 범위에서 제외한다. 서버 API, D1 스키마, 계정 인계·복구 코드 규칙도 변경하지 않는다.

## 관리 화면 컨트롤

공통 CSS 클래스와 필요 시 작은 재사용 UI 컴포넌트를 사용한다. 브라우저 기본 요소인 `input`, `select`, `checkbox`는 유지한다. 따라서 폼 제출, 키보드 조작, 스크린리더 의미와 기존 테스트의 접근성 이름을 보존한다.

- 인라인 텍스트 입력은 표 안에서도 충분한 높이·여백·테두리·포커스 링을 가진다.
- 역할 선택은 동일한 높이와 둥근 테두리, 디자인에 맞는 화살표를 사용한다.
- 활성 상태와 담당 조직 선택은 시각적으로 명확한 커스텀 체크 컨트롤을 사용한다. 활성 상태 라벨은 현재 값에 따라 `사용` 또는 `중지`를 유지한다.
- 포커스, 비활성화, 호버 상태는 기존 색상 토큰과 대비 규칙을 따른다.

## 비밀번호 확인 흐름

`ChangePasswordPage`와 `RecoveryPage`는 `새 비밀번호` 바로 다음에 `새 비밀번호 확인` 필드를 렌더링한다.

1. 제출 시 새 비밀번호를 기존 `PasswordSchema`로 검증한다.
2. 두 필드가 다르면 `새 비밀번호가 일치하지 않습니다.`를 보여 주고 네트워크 요청을 보내지 않는다.
3. 두 필드가 같을 때만 기존 비밀번호 변경 또는 복구 API를 호출한다.
4. 각 필드 변경 시 기존 검증 오류를 제거한다. 두 필드는 `type="password"`, `required`, `autoComplete="new-password"`를 사용한다.

## 검증

- 관리 화면 테스트는 역할 변경, 조직 선택, 활성 상태 변경의 기존 요청 본문을 계속 검증한다.
- 인증 화면 테스트는 비밀번호 불일치가 API 호출을 차단하는지, 일치 시 기존 요청 본문으로 전송하는지 확인한다.
- 웹 테스트, 타입 검사, 포맷 검사, 웹 빌드, 로컬 브라우저 E2E를 실행한다.
