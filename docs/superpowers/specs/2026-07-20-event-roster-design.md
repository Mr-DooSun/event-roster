# Event Roster MVP 설계

- 작성일: 2026-07-20
- 상태: 사용자 승인 완료
- 대상: 상반기·하반기에 각각 한 번 운영하는 내부 행사 참가자 명단 서비스

## 1. 목표와 범위

이 서비스는 행사 전 조직별 예상 참가자 명단과 행사 당일의 추가·취소·수정을 한 곳에서 관리한다. 운영자는 조직별·전체 예상 인원, 당일 변동, 최종 유효 인원을 즉시 확인하고 행사별 명단을 엑셀로 내보낼 수 있어야 한다.

MVP에는 다음을 포함한다.

- 행사 생성, 상태 전환, 마감·재오픈
- 조직별 사전 참가자 등록
- 당일 참가자 추가·취소·수정
- 조직별·전체 예상 및 최종 인원 집계
- 반복 행사에서 재사용하는 참가자 마스터
- 조직별 권한, 변경 감사 이력, 동시 수정 충돌 방지
- 기존 엑셀의 브라우저 기반 이관과 행사별 엑셀 내보내기
- 운영자 생성·재설정 방식의 자체 계정과 JWT 인증

MVP에서 제외한다.

- 현장 체크인과 실제 출석 수 집계
- 원본 엑셀 파일 보관
- 공개 회원가입, 셀프 비밀번호 찾기, 이메일 발송
- 엑셀 파일과 서비스 데이터의 양방향 동기화
- 실시간 공동 편집, 채팅, 알림 자동 발송
- 행사 종료 뒤의 삭제 또는 재사용

`실제 인원`은 체크인 수가 아니라 당일 변경이 반영된 최종 유효 명단 수를 뜻한다.

## 2. 핵심 결정

| 항목 | 결정 |
| --- | --- |
| 행사 주기 | 상반기·하반기 각 1회 |
| 명단 소유 | 모든 명단은 특정 행사에 귀속 |
| 참가자 식별 | 시스템이 발급한 참가자 번호 + 이름 + 현재 소속 조직 |
| 참가자 재사용 | 행사와 독립된 참가자 마스터에서 검색·선택 |
| 수정 권한 | 운영자는 전체, 조직 담당자는 자기 조직만 |
| 변경 이력 | 작업자·시각·변경 유형·전후 값까지 영구 보관 |
| 실제 인원 | 당일 유효 명단 수 |
| 예상 인원 | 당일 운영 전환 순간의 조직별 스냅샷 |
| 엑셀 | 원본 파일 미보관, 브라우저에서 변환·검토, 서비스는 데이터만 저장 |
| 인증 | 운영자 생성 계정, 비밀번호 로그인, JWT HttpOnly 쿠키 |
| 배포 | React/Vite Pages + FastAPI Python Worker + D1, 모노레포 |

## 3. 사용자, 계정, 권한

### 운영자

- 행사·조직·사용자 계정을 생성하고 수정한다.
- 모든 조직과 모든 행사 명단을 조회·수정한다.
- 행사 상태를 전환하고, 마감된 행사를 재오픈한다.
- 엑셀 이관·내보내기와 감사 이력 조회를 수행한다.
- 조직 담당자의 초기 비밀번호를 설정하거나 재설정한다.

### 조직 담당자

- 연결된 자기 조직의 참가자 마스터를 검색하고 새 참가자를 등록한다.
- 자기 조직의 열려 있는 행사 명단만 추가·취소·수정한다.
- 자기 조직의 조직별 집계와 변경 이력을 조회한다.
- 다른 조직의 참가자·명단·감사 이력은 조회하거나 수정할 수 없다.
- 조직 담당자는 참가자의 이름을 정정할 수 있지만, 참가자 마스터의 현재 소속을 다른 조직으로 옮길 수는 없다. 조직 이동은 운영자만 수행한다.

### 인증

- 공개 가입은 없다. 운영자가 이메일, 표시 이름, 역할, 소속 조직을 지정해 계정을 만든다.
- 로그인 성공 후 서버는 짧은 수명의 JWT를 `HttpOnly`, `Secure`, `SameSite` 쿠키로 발급한다. 프론트엔드는 토큰을 `localStorage`에 저장하지 않는다.
- 로그아웃, 비밀번호 재설정, 계정 비활성화는 활성 세션을 폐기한다.
- 로그인은 IP와 이메일 기준의 실패 횟수 제한을 적용하고, 상태를 바꾸는 API는 허용된 Origin과 CSRF 방어를 확인한다.
- 자체 비밀번호 해싱은 Cloudflare Python Worker 호환성 및 Free CPU 한도를 만족해야 한다. 이를 충족하지 못하면 해싱 강도를 낮추지 않고 배포·인증 설계를 재검토한다.

## 4. 행사 수명주기

행사는 독립된 최상위 관리 단위다. 기본 제목은 예를 들어 `2026 상반기 행사`이며, 운영자가 제목·행사일·장소를 입력한다. 장소는 선택값이다. 같은 연도·반기에는 하나의 행사만 만들 수 있다. 기본 목록은 초안·사전 등록·당일 운영 행사만 보여 주고, 마감 행사는 `지난 행사`에서 읽기 전용으로 조회한다.

```text
초안 → 사전 등록 → 당일 운영 → 마감
                         ↑         │
                         └─ 재오픈 ┘
```

| 상태 | 변경 권한과 동작 |
| --- | --- |
| 초안 | 운영자만 행사 기본 정보를 설정한다. 명단은 아직 편집할 수 없다. |
| 사전 등록 | 운영자와 조직 담당자가 권한 범위에서 명단을 갱신한다. 예상 인원은 현재 유효한 사전 명단 수다. |
| 당일 운영 | 운영자가 전환하는 즉시 조직별 예상 인원 스냅샷을 생성한다. 이후 추가·취소·수정은 변동과 최종 인원에 반영한다. |
| 마감 | 읽기 전용이다. 집계·감사 이력·엑셀 내보내기만 허용한다. |
| 재오픈 | 운영자만 `당일 운영` 상태로 되돌릴 수 있다. 기존 예상 인원 스냅샷은 유지하며 재오픈 사실을 감사 이력에 남긴다. |

사전 등록으로 되돌리는 전이는 제공하지 않는다. 이렇게 해야 당일 비교 기준이 바뀌지 않는다.

## 5. 데이터 모델

### 조직과 사용자

- `organizations`: 조직 이름, 활성 여부
- `users`: 이메일, 표시 이름, 역할(`OPERATOR`, `ORGANIZATION_MANAGER`), 비밀번호 해시, 활성 여부, 세션 버전
- `user_organizations`: 조직 담당자와 조직의 연결. 현재 한 조직 담당자가 여러 조직을 맡을 수 있도록 관계 테이블로 둔다.
- `auth_sessions`: JWT의 세션 식별자, 만료 시각, 폐기 시각을 보관해 로그아웃·계정 비활성화·비밀번호 재설정 후 세션을 무효화한다.

### 참가자와 행사

- `participants`: 시스템 참가자 번호, 이름, 현재 조직, 활성 여부, 생성·수정 시각
- `events`: 제목, 연도, 반기(`H1`, `H2`), 행사일, 선택 장소, 상태, 상태 전환 시각. `(연도, 반기)`는 고유하다.
- `event_roster_entries`: 행사와 참가자를 연결하는 명단 행
  - 참가자 번호 참조
  - 행사 당시 이름과 조직 ID·조직 이름 스냅샷
  - 등록 출처(`PRE_EVENT`, `DAY_OF`)
  - 상태(`ACTIVE`, `CANCELLED`)
  - 낙관적 잠금용 `revision`
  - 생성·수정 시각
- `event_expected_snapshots`: 당일 운영 전환 시점의 행사·조직별 예상 유효 인원과 캡처 시각

각 행사는 참가자당 명단 행을 하나만 가진다. `(event_id, participant_id)`를 고유하게 두고, 취소 뒤 재참여는 새 행을 만들지 않고 기존 행을 재활성화한다.

조직 이동 또는 참가자 이름 수정은 참가자 마스터의 현재 값만 바꾼다. 이미 마감된 행사 명단은 당시 이름·조직 스냅샷을 계속 보인다. 현재 행사에서 참가자 기본 정보를 고칠 때는 마스터와 현재 행사 명단 스냅샷을 함께 수정하되, 과거 행사에는 영향을 주지 않는다. 조직 담당자는 자기 조직 안에서 이름만 정정할 수 있으며, 참가자의 현재 소속을 바꾸는 작업은 운영자 전용이다.

취소된 명단 행은 절대 삭제하지 않는다. `CANCELLED`로 전환해 집계와 감사 이력에서 추적한다.

### 감사와 이관 기록

- `audit_logs`: 작업자, 행사, 명단 행 또는 대상 엔터티, 변경 유형, 전 값, 후 값, 발생 시각
- `import_runs`: 행사, 실행 사용자, 처리 행 수, 생성·갱신·오류 수, 확정 시각. 원본 파일과 원본 셀 값은 저장하지 않는다.

감사 이력은 수정·삭제 API를 제공하지 않는 append-only 데이터다. 명단 변경과 그 감사 이력은 하나의 원자적 작업으로 저장한다.

## 6. 인원 계산 규칙

| 시점 | 예상 인원 | 최종 인원 |
| --- | --- | --- |
| 사전 등록 | 현재 `PRE_EVENT` + `ACTIVE` 명단 수 | 동일 수를 참고값으로 표시 |
| 당일 운영 | `event_expected_snapshots`의 고정 수 | 모든 `ACTIVE` 행사 명단 행 수 |
| 마감 | 당일 전환 시 고정된 수 | 마감 시점의 모든 `ACTIVE` 행사 명단 행 수 |

조직별 변동은 예상 기준값과 최종 인원의 차이로 표시한다. 추가·취소 건수는 감사 이력과 등록 출처를 기반으로 함께 보여 준다.

## 7. 화면과 디자인 패턴

기본 패턴은 **하이브리드 운영 콘솔**이다. 행사당 70~130명, 동시 운영자 약 13명이라는 규모에 맞춰 카드만으로 명단을 표현하지 않는다.

### 행사 운영 화면

- 상단: 행사명, 상태 배지, 행사일, 상태 전환 행동
- 요약: 전체 예상·최종·변동, 조직별 변화가 큰 항목
- 본문: 고정 헤더를 가진 명단 테이블. 이름·조직·상태·등록 출처·최종 수정 시각을 표시하고 조직·상태·이름으로 필터한다.
- 오른쪽 패널: 선택 참가자의 현재 행사 정보, 빠른 추가·취소·수정, 최근 변경 이력
- 당일 운영 모드: 현재 행사와 조직별 변동, 빠른 명단 수정, 최근 변경을 우선 노출한다.

70~130행은 가상 스크롤이나 페이지 나누기 없이 한 테이블에 표시한다. 데스크톱 사용을 우선하되, 좁은 화면에서는 읽기·조회 중심의 카드 표현으로 전환한다.

### 가져오기 화면

가져오기는 일반 탐색 화면을 잠시 숨기는 집중형 4단계 흐름이다.

1. 브라우저에서 엑셀 파일과 시트를 선택한다.
2. 이름·조직 열을 매핑하고 샘플 행을 미리본다.
3. 빈 필수값, 알 수 없는 조직, 같은 이름·조직의 중복, 마스터 후보가 여러 개인 행을 해결한다.
4. 오류가 0개일 때만 한 번에 확정한다.

확정 전에 오류 행을 건너뛰거나 부분 저장하지 않는다. API는 확정 요청에서 권한·행사 상태·조직 범위·중복을 다시 검증한다.

### 디자인 시스템

`coursemos-supporter`의 디자인 문서에서 다음 원칙을 가져온다.

- 색상, 간격, 라운드, 그림자, 글자 굵기는 `--er-*` CSS 토큰으로 관리한다.
- `Button`, `Badge`, `Dialog`, `Tabs`, `Table`, `EmptyState`, `Progress` 같은 작은 UI 프리미티브를 공통화한다.
- 화면별 도메인 로직은 프리미티브를 조합하고, 시각 표현은 토큰과 CSS가 담당한다.
- 빈 상태, 처리 중, 검증 오류, 충돌, 완료 상태마다 다음 행동을 명확히 제시한다.

참조 프로젝트의 카드 중심 사이드패널을 그대로 복제하지 않는다. 카드와 집중형 흐름은 대시보드·가져오기에 쓰고, 명단 주 화면은 정보 밀도가 높은 테이블을 쓴다.

## 8. 모노레포와 코드 경계

```text
event-roster/
├── apps/
│   ├── web/                 # React/Vite, Cloudflare Pages
│   └── api/                 # FastAPI Python Worker
├── packages/
│   └── contracts/           # OpenAPI에서 생성한 API 타입과 공유 계약
└── docs/
```

프론트엔드는 기능 중심 구조를 사용한다.

```text
apps/web/src/
├── app/            # 라우팅, 인증 진입점, 전역 레이아웃
├── components/ui/  # 공통 UI 프리미티브
├── features/       # events, roster, participants, import, audit-log
└── lib/            # API 클라이언트와 공통 유틸리티
```

백엔드는 과도한 추상화를 피한 얇은 모듈형 모놀리스다.

```text
apps/api/app/
├── api/             # HTTP 라우터, 요청·응답 DTO
├── application/     # 행사·명단·이관 유스케이스
├── domain/          # 상태 전이, 권한, 집계, 충돌 규칙
└── infrastructure/  # D1, JWT, Cloudflare Worker 연동
```

라우터는 HTTP 변환만 담당하고, 유스케이스는 권한·상태·트랜잭션 경계를 확인한 뒤 도메인 규칙을 호출한다. D1의 구체적인 호출과 JWT·Worker API 접근은 `infrastructure`에 둔다. 단일 D1 구현만으로 충분한 MVP에서는 인터페이스를 위한 인터페이스를 만들지 않는다.

## 9. 런타임과 배포

- `apps/web`은 React/Vite를 정적 빌드해 Cloudflare Pages에 배포한다. Node 서버를 상시 실행하지 않는다.
- `apps/api`는 Cloudflare Python Worker의 ASGI 어댑터로 FastAPI를 실행한다.
- D1은 사용자, 행사, 명단, 스냅샷, 감사 이력을 저장한다.
- 웹과 API는 같은 사이트 도메인 구성으로 배포한다. API는 정확한 웹 origin만 CORS 허용하고 credential 요청만 수락한다.
- 엑셀 원본은 브라우저에서만 읽고 보관하지 않는다. 브라우저가 정규화한 행 데이터를 API로 전송한다.

Cloudflare Free 한도와 Python Worker의 패키지 제약 때문에, 구현 시작 전 다음 기술 검증을 먼저 통과해야 한다.

1. 보안 기준을 충족하는 비밀번호 해싱과 검증이 Python Worker에서 동작한다.
2. JWT 발급·검증·세션 폐기가 D1과 함께 동작한다.
3. 행사 명단 갱신과 감사 이력이 실패 시 함께 롤백된다.
4. 130행 이관·조회·내보내기 흐름이 Worker CPU·메모리 한도 내에서 동작한다. 엑셀 파싱·생성은 브라우저에서 수행한다.

위 검증에서 보안 또는 한도 조건을 만족하지 못하면 약한 암호화나 부분 저장으로 우회하지 않는다. 배포 또는 인증 방식을 다시 설계한다.

## 10. API와 주요 데이터 흐름

### 행사와 명단

- 운영자는 행사 생성 후 `사전 등록`으로 전환한다.
- 조직 담당자는 자기 조직의 참가자를 검색·등록하고 해당 행사 명단에 추가한다.
- 명단 저장 요청은 현재 `revision`을 포함한다. 서버의 최신 revision과 다르면 `409 Conflict`로 최신 행과 충돌 정보를 반환한다.
- 운영자가 `당일 운영`으로 전환하면 서버가 조직별 예상 인원 스냅샷을 생성한다.
- 당일 추가·취소·수정은 행사 명단과 감사 이력을 함께 갱신한다.
- 운영자가 마감하면 수정 요청은 거부한다. 재오픈은 운영자만 할 수 있으며 `당일 운영`으로 되돌린다.

### 엑셀 이관

- 브라우저가 워크북을 읽고 사용자가 이름·조직 열을 매핑한다.
- 브라우저는 오류와 중복 후보를 보여 주지만, 확정 전 데이터는 저장하지 않는다.
- 확정 요청은 정규화된 행 목록과 행사 식별자를 보낸다.
- 서버는 다시 조직 권한, 행사 상태, 필수값, 중복, 낙관적 잠금을 검증한다.
- 모든 행이 통과할 때만 참가자 마스터·행사 명단·감사 이력·이관 실행 기록을 함께 저장한다.

### 엑셀 내보내기

- 내보내기는 선택한 행사 기준이다.
- `명단` 시트: 참가자 번호, 이름, 행사 당시 조직, 등록 출처, 상태, 최종 수정 시각
- `집계` 시트: 조직별 사전 예상, 당일 추가, 당일 취소, 최종 유효 인원
- 브라우저에서 `.xlsx` 파일을 만든다. 원본 업로드 파일이나 다른 행사의 데이터는 포함하지 않는다.

## 11. 오류 처리와 보안 규칙

| 상황 | 처리 |
| --- | --- |
| 로그인 실패 | 계정 존재 여부를 노출하지 않는 동일한 오류 메시지 |
| 권한 없는 조직 접근 | `403`으로 거부하고 감사 보안 로그를 남김 |
| 마감 행사 수정 | `409`으로 거부하고 마감 상태를 반환 |
| 오래된 명단 저장 | `409`과 최신 행·변경자·수정 시각을 반환, 사용자에게 병합 또는 새로고침을 안내 |
| 엑셀 검증 오류 | 행 번호·원인·수정 방법을 반환, 저장하지 않음 |
| 이관 중 서버 실패 | 전체 작업을 실패 처리하고 부분 명단·부분 감사 이력을 남기지 않음 |
| 데이터베이스·Worker 오류 | 사용자에게는 재시도 가능한 일반 메시지, 내부에는 원인과 요청 상관관계 기록 |

비밀번호 값과 JWT 원문은 감사 로그·애플리케이션 로그·엑셀 내보내기에 절대 포함하지 않는다.

## 12. 테스트 전략과 완료 기준

### 도메인·유스케이스 테스트

- 역할과 조직 범위에 따른 허용·거부
- 행사 상태 전이와 재오픈 기록
- 사전 예상 스냅샷 생성과 조직별·전체 최종 인원 계산
- 취소 행 보존과 과거 조직 스냅샷 보존
- 낙관적 잠금 충돌
- 명단 변경과 감사 이력의 원자성

### API·인프라 통합 테스트

- FastAPI Worker ASGI 경로와 D1 바인딩
- 자체 계정 로그인, JWT 쿠키, 로그아웃·비밀번호 재설정 후 세션 폐기
- 조직 담당자의 교차 조직 접근 거부
- 130행 엑셀 이관 확정의 전체 성공과 단일 오류 시 전체 실패

### 프론트엔드·브라우저 테스트

- 행사 생성·상태 전환·명단 테이블 필터
- 빠른 추가·취소·수정 패널과 충돌 안내
- 엑셀 열 매핑·오류 검토·확정 버튼 비활성화
- 명단·집계 시트가 포함된 엑셀 내보내기
- 빈 상태, 로딩, 오류, 완료 상태의 접근 가능한 행동

### 인수 기준

1. 운영자는 새 행사를 만들고 사전 등록, 당일 운영, 마감, 재오픈을 순서대로 수행할 수 있다.
2. 조직 담당자는 자기 조직의 명단만 수정하고, 다른 조직의 데이터에는 접근할 수 없다.
3. 당일 전환 뒤 사전 예상 수는 고정되고, 추가·취소가 최종 수에 반영된다.
4. 두 사용자가 같은 명단 행을 수정하면 먼저 저장된 값이 보존되고 두 번째 사용자는 충돌을 해결해야 한다.
5. 엑셀 이관은 오류가 없는 경우에만 전부 반영되고, 원본 파일은 저장되지 않는다.
6. 마감된 행사도 과거 명단·집계·감사 이력·엑셀 내보내기를 제공한다.
7. Cloudflare Free 환경에서 인증·D1·130행 명단 흐름의 기술 검증을 통과한 뒤 기능 구현을 시작한다.
