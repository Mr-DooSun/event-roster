# 비동기 로딩 피드백 설계

## 목표

느린 네트워크 또는 API 응답을 기다리는 동안 사용자가 현재 상태를 명확히
이해하게 한다. 최초 조회, 기존 데이터 재조회, 생성·수정·삭제, 엑셀 처리에
각각 알맞은 피드백을 제공하되 화면 전체를 불필요하게 차단하지 않는다.

이 설계는 기다리는 경험을 개선하는 프런트엔드 작업이다. Worker, D1 쿼리,
API 호출 횟수와 응답 시간 자체를 측정하거나 최적화하는 작업은 포함하지
않는다.

## 범위

- 프로젝트 목록과 프로젝트 생성
- 프로젝트 상세 공통 정보
- 프로젝트 개요, 연결 조직, 참가 명단, 변경 이력
- 조직 관리 목록과 조직 상세
- 계정 관리
- 로그인, 비밀번호 변경, 부트스트랩 계정 인계
- 엑셀 가져오기와 내보내기
- 공통 버튼, 스켈레톤, 로딩 상태, 재시도 오류 UI

백엔드 API 계약, 서버 데이터 처리 방식, 낙관적 업데이트, 전역 상태관리
라이브러리와 데이터 조회 라이브러리는 추가하거나 변경하지 않는다.

## 선택한 접근

공통 로딩 컴포넌트를 만들고 각 화면이 자신의 실제 레이아웃에 맞게
조합한다.

전역 로딩 오버레이는 로그인 세션 복원처럼 결과에 따라 렌더링할 화면
자체가 달라지는 경우에만 사용한다. 프로젝트, 조직, 계정, 참가 명단 요청은
관련된 콘텐츠 영역에서 상태를 표시한다. 모든 화면이 독립적으로 로딩 UI를
구현하는 방식은 문구, 애니메이션, 접근성과 테스트가 중복되므로 사용하지
않는다.

기존 수동 요청 상태와 오래된 응답 방지용 generation 패턴은 유지한다.
TanStack Query 같은 새 의존성이나 범용 비동기 상태 훅은 도입하지 않는다.

## 상태 모델

### 조회

조회 영역은 다음 상태를 구분한다.

- `idle`: 아직 요청을 시작하지 않은 상태
- `initial-loading`: 표시할 데이터 없이 최초 응답을 기다리는 상태
- `success`: 하나 이상의 결과를 표시하는 상태
- `empty`: 요청은 성공했지만 결과가 없는 상태
- `error`: 최초 조회가 실패해 표시할 데이터가 없는 상태
- `refreshing`: 기존 데이터를 유지한 채 최신 응답을 기다리는 상태

빈 상태는 요청 성공이 확인된 뒤에만 표시한다. 최초 조회 전의 빈 배열을
데이터 없음으로 해석하지 않는다.

재조회에 실패했지만 기존 데이터가 있으면 데이터를 보존하고 비차단 오류
메시지와 `다시 시도`를 표시한다. 최초 조회 실패는 스켈레톤을 제거하고
해당 영역을 오류 상태로 교체한다.

### 변경

생성·수정·삭제 요청은 다음 상태를 구분한다.

- `idle`: 요청 가능한 상태
- `submitting`: 요청을 보냈으며 응답을 기다리는 상태
- `error`: 요청이 실패하고 사용자가 다시 시도할 수 있는 상태

클릭 즉시 관련 버튼 또는 행을 비활성화하고 진행 문구를 표시한다. 실제
데이터는 서버 성공 응답 후 반영한다. 실패한 폼의 입력값은 유지한다.
화면 전체나 관련 없는 버튼은 차단하지 않는다.

동일 요청의 중복 제출을 막되, 서로 독립적인 영역의 조회 상태와 오류는
서로 덮어쓰지 않는다.

## 공통 컴포넌트

### `Skeleton`

카드, 텍스트, 배지, 표 행처럼 실제 콘텐츠의 크기와 배치를 대략 유지하는
표시 전용 요소다. 형태와 너비를 class 또는 prop으로 조합할 수 있게 하고,
데이터 의미를 모방하는 가짜 텍스트는 넣지 않는다.

스켈레톤 자체는 `aria-hidden="true"`로 보조기술에서 숨긴다. 스켈레톤을
포함하는 영역은 `aria-busy="true"`를 사용하고 별도의 상태 문구를
제공한다.

### `LoadingStatus`

`불러오는 중…`, `새로고침 중…`, `더 불러오는 중…`처럼 실제 진행 상태를
짧은 한국어 문구로 전달한다. 시각적으로는 작은 스피너와 문구를 제공하고,
상태 컨테이너에 `role="status"`와 `aria-live="polite"`를 적용한다.

초기 스켈레톤과 함께 사용할 때는 화면 낭독기용 문구 역할을 하고, 기존
데이터 재조회나 페이지네이션에서는 화면에도 보이는 인라인 상태로
사용한다.

### `Button`

기존 `Button`에 `loading?: boolean`과 `loadingText?: string`을 추가한다.
`loading`이 참이면 버튼을 비활성화하고 스피너와 `loadingText`를
렌더링한다. 문구를 생략한 호출부는 범용 `처리 중…`을 사용하지만, 주요
작업은 `저장 중…`, `만드는 중…`, `내보내는 중…`처럼 동작을 명시한다.

버튼의 기존 variant와 type 동작은 유지한다. 로딩 문구로 바뀌어도 버튼
폭이 과도하게 흔들리지 않게 현재 크기 체계 안에서 표시한다.

### `RetryableError`

영역별 오류 문구와 `다시 시도` 버튼을 함께 제공한다. 재시도 중에는 버튼의
로딩 상태를 사용해 중복 요청을 막는다. 기존 `StatusMessage`의 색상과
의미를 재사용하고, 폼 validation 오류처럼 재요청으로 해결되지 않는
오류에는 사용하지 않는다.

## 화면별 동작

### 인증

세션 복원은 현재 전체 화면 구조를 유지하고 공통 `LoadingStatus`를
사용한다. 로그인, 비밀번호 변경, 복구와 부트스트랩 인계는 제출 버튼에
진행 문구를 표시하며 입력 폼 전체를 제거하지 않는다.

### 프로젝트 목록

최초 조회에는 실제 프로젝트 그리드와 같은 열 규칙을 사용하는 프로젝트
카드 스켈레톤을 표시한다. 고정된 여섯 개 카드를 사용해 데스크톱과
모바일의 레이아웃 이동을 줄인다.

조회 성공 후 결과가 없을 때만 프로젝트 없음 상태를 표시한다. 프로젝트
생성 중에는 제출 버튼을 `만드는 중…`으로 바꾸고 성공 후 목록을 다시
불러오는 동안 기존 목록을 유지한다.

### 프로젝트 상세

프로젝트 이름, 상태, 기간과 상단 동작 영역의 스켈레톤을 먼저 표시한다.
프로젝트 자체를 불러오지 못하면 상세 전체를 `RetryableError`로 바꾼다.

요약, 연결 조직, 참가 명단, 참가자 후보와 변경 이력은 현재처럼 병렬로
요청하되 영역별 로딩과 오류를 독립적으로 관리한다. 프로젝트 셸을
불러온 뒤에는 응답이 끝난 영역부터 사용할 수 있고, 느리거나 실패한
영역이 다른 탭의 데이터를 가리지 않는다.

같은 탭 안에서도 참가자 후보는 참가 명단 표와, 전체 조직 후보는 프로젝트
연결 조직 목록과 별도의 경계를 사용한다. 후보 조회가 끝나지 않았거나
실패한 경우 성공한 표·목록은 유지하고 후보가 필요한 추가·수정 동작만
비활성화한다. 프로젝트 header 재시도는 `/projects/:id`만 재조회하며 다른
상세 resource와 audit pagination을 초기화하지 않는다.

프로젝트 상태 변경, 일정 수정과 이름 수정은 실행한 버튼 또는 모달의
제출 버튼만 잠근다. 상태 변경 후 재조회하는 동안 기존 프로젝트 셸을
유지한다. 상태 변경 뒤 project shell 요청과 detail resource refresh는 함께
시작하지만, 최신 project shell이 성공하면 느린 detail resource와 무관하게
header 동작 잠금을 해제한다. project shell 자체가 pending 또는 failed이면
이전 revision 동작은 계속 잠근다.

### 연결 조직과 참가 명단

조직 목록은 카드 또는 행 구조의 스켈레톤을 사용한다. 조직 추가, 연결
상태 수정은 관련 폼과 버튼에만 진행 상태를 표시한다.

참가 명단은 표 헤더를 유지하고 여러 개의 행 스켈레톤을 표시한다. 참가자
추가·수정·취소는 서버 성공 후 실제 행에 반영하며, 수행 중인 모달이나
행의 동작만 비활성화한다.

엑셀 내보내기는 버튼을 `내보내는 중…`으로 표시한다. 내보내기 실패는
기존 명단을 유지하고 다시 실행할 수 있게 한다.

### 변경 이력

최초 조회에는 이력 행 스켈레톤을 사용한다. 추가 페이지 조회 중에는 기존
이력을 유지하고 목록 아래에 `더 불러오는 중…`을 표시한다. 실패 시 기존
이력을 유지하면서 페이지네이션 영역에 재시도 버튼을 제공한다.

### 조직·계정 관리

조직과 계정의 최초 목록 조회에는 실제 카드 또는 표에 맞는 스켈레톤을
표시한다. 조직 검색과 필터 변경은 기존 결과를 유지하고 검색 영역에
`새로고침 중…`을 표시한다.

조직 상세의 기본 정보, 담당자와 연결 프로젝트는 독립적인 영역 상태를
사용한다. 조직·계정 생성, 수정, 비밀번호 초기화와 사용 상태 변경은 해당
버튼 또는 폼에 진행 문구를 표시한다.

### 엑셀 가져오기

현재 단계와 사용자가 선택한 파일, 열 매핑, 충돌 해결 입력은 요청 중에도
유지한다. 검증은 `검증 중…`, 반영은 `가져오는 중…`으로 구분하고 관련
단계의 입력과 버튼만 비활성화한다. 기존 `busy` 의미를 공통 버튼과 진행
상태 표현에 연결한다.

## 오류와 동시성

- 요청 시작과 동시에 상태를 갱신하며 인위적인 최소 로딩 시간은 두지
  않는다.
- 오래된 요청이 나중에 성공하거나 실패해도 최신 화면과 메시지를
  덮어쓰지 않는다.
- 재시도는 실패한 영역의 요청만 다시 실행한다.
- 인증 만료와 권한 오류는 기존 `AuthProvider`와 API 오류 처리 흐름을
  유지한다.
- 동시 수정 충돌, 종료된 프로젝트와 validation 오류의 기존 전용 문구를
  일반 네트워크 오류로 바꾸지 않는다.
- 컴포넌트가 unmount되거나 다른 프로젝트로 이동하면 이전 요청의 상태
  반영을 무시한다.

## 시각 디자인과 반응형

- 스켈레톤은 기존 surface, border, radius와 spacing 토큰을 사용한다.
- 강조 애니메이션은 낮은 대비의 단일 shimmer 또는 pulse만 사용한다.
- `prefers-reduced-motion: reduce`에서는 애니메이션을 제거한다.
- 카드, 표, 모달의 스켈레톤은 실제 콘텐츠와 비슷한 높이를 확보해 응답
  전후 레이아웃 이동을 줄인다.
- 모바일에서는 기존 콘텐츠와 동일한 열 전환 규칙을 사용하며 가로
  overflow를 새로 만들지 않는다.
- 전체 화면 로딩은 인증 복원 외에는 추가하지 않는다.

## 접근성

- 로딩 중인 콘텐츠 영역은 `aria-busy="true"`를 사용한다.
- 진행 상태 문구는 `aria-live="polite"`로 알리고 스켈레톤 모양은
  `aria-hidden="true"`로 숨긴다.
- 로딩 버튼은 native `disabled`를 사용해 중복 실행과 focus 재진입을
  막고 화면에 동작별 진행 문구를 표시한다.
- 오류 메시지와 재시도 버튼은 키보드로 접근 가능해야 한다.
- 색상이나 애니메이션만으로 로딩과 오류를 표현하지 않는다.

## 테스트

- 지연된 Promise가 끝나기 전 최초 스켈레톤이 표시되고 빈 상태는
  표시되지 않는다.
- 결과가 하나 이상이면 콘텐츠로, 성공한 빈 결과면 빈 상태로 전환한다.
- 최초 조회 실패는 영역 오류와 `다시 시도`를 표시하고 재시도 성공 시
  콘텐츠로 전환한다.
- 재조회 중 기존 데이터를 유지하고 완료 후 새 데이터로 교체한다.
- 재조회 실패 시 기존 데이터와 재시도 동작을 유지한다.
- 제출 중 버튼이 비활성화되고 동작별 문구를 표시하며 중복 요청을 보내지
  않는다.
- 제출 실패 시 입력값을 유지하고 다시 제출할 수 있다.
- 프로젝트 상세의 병렬 요청은 영역별로 독립적인 로딩, 성공과 오류를
  표시한다.
- 이전 프로젝트 또는 이전 필터 요청의 늦은 응답이 최신 결과를 덮지
  않는다.
- 이력 추가 조회와 재시도는 기존 목록을 보존한다.
- `aria-busy`, live 상태 문구와 스켈레톤의 숨김 속성을 검증한다.
- `prefers-reduced-motion` 스타일과 데스크톱·모바일 레이아웃을
  확인한다.
- 전체 Web 테스트, type check와 production build에서 회귀가 없어야 한다.

## 완료 기준

- 사용자는 모든 주요 조회에서 데이터 없음과 로딩 중을 구분할 수 있다.
- 최초 조회 시 실제 화면 형태와 유사한 스켈레톤을 볼 수 있다.
- 재조회 중 기존 데이터가 사라지지 않는다.
- 모든 주요 변경 작업은 중복 실행을 막고 무엇을 처리 중인지 표시한다.
- 일시적인 조회 실패는 실패한 영역에서 바로 재시도할 수 있다.
- 한 영역의 지연이나 실패가 사용 가능한 다른 영역을 차단하지 않는다.
- 키보드와 화면 낭독기 사용자는 로딩, 완료와 오류 상태를 이해할 수 있다.
- 기존 API 계약, 권한, 감사 이력과 동시 수정 규칙은 그대로 유지된다.
