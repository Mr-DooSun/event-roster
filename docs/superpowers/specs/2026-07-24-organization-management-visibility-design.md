# 조직 담당자 지정 모달과 조직 요약 시인성 개선 설계

## 1. 배경

조직 상세의 `기존 담당자 지정` 모달은 검색, 계정 선택, 역할 선택,
확정 동작이 같은 수준의 그리드에 배치되어 작업 순서와 구분감이 약하다.
조직 목록 카드의 요약값도 레이블과 값에서 같은 의미를 반복해 좁은
카드에서 줄바꿈이 발생한다.

조직 목록, 조직 상세, 프로젝트 조직 화면은 대표 조직장과 추가 관리자를
서로 다른 표현으로 보여 주고 있다. 사용자가 실제로 조직 명단을 관리할
수 있는 사람이 모두 몇 명인지 바로 파악하기 어렵다.

## 2. 목표

- 기존 담당자 지정 흐름을 `계정 찾기`와 `담당 범위 설정` 두 단계로
  시각적으로 분리한다.
- 조직 요약값을 짧고 일관된 표현으로 바꾼다.
- `담당자`를 대표 조직장을 포함한 전체 담당 인원으로 정의한다.
- 데스크톱과 모바일에서 입력 요소와 동작 버튼이 명확히 구분되게 한다.
- 기존 검색, 지정, 역할, 권한, 감사 동작은 그대로 유지한다.

## 3. 범위

### 포함

- `기존 담당자 지정` 모달의 레이아웃과 안내 문구
- 조직 목록 카드의 정보 칸
- 조직 상세 상단 요약
- 프로젝트 상세의 조직 카드 요약
- 전체 담당자 수를 계산하는 프런트 공통 함수
- 관련 컴포넌트·접근성·회귀 테스트

### 제외

- Worker API, D1 schema, 계약 타입 변경
- 대표 조직장과 추가 관리자의 권한 변경
- 새 담당자 발급 모달의 입력 흐름 변경
- 조직 담당자 배정·해제 감사 이벤트 변경
- 조직 목록 필터와 검색 조건 변경

## 4. 선택한 방향

시각 비교안 중 A안인 `구획형 흐름`을 사용한다.

기존 모달처럼 두 form을 같은 그리드로 연속 배치하지 않는다. 검색과
지정을 각각 옅은 배경과 테두리를 가진 섹션으로 만들고, 번호가 있는
단계 제목을 제공한다. 단순 구분선형보다 작업 순서가 명확하고,
컴팩트형보다 입력과 동작 버튼이 잘 구분된다.

조직 목록 카드는 세 요약값을 독립된 정보 칸으로 표시한다. 조직 상세
상단과 프로젝트 조직 카드는 기존 공간 구조를 유지하되 같은 문구와
숫자 의미를 사용한다.

## 5. 기존 담당자 지정 모달

### 5.1 구조

모달 제목은 `기존 담당자 지정`을 유지한다. 모달 본문은 다음 순서다.

1. `계정 찾기`
   - 검색 필드 레이블: `로그인 ID 또는 표시 이름`
   - 검색 버튼: `검색`
   - 검색 진행 문구: `검색 중…`
2. `담당 범위 설정`
   - `지정할 계정` select
   - `조직별 역할` select
3. 하단 action 영역
   - 보조 동작: `취소`
   - 주요 동작: `담당자로 지정`

각 단계는 독립된 시각 영역이다. 단계 제목은 번호 배지와 텍스트를 함께
사용하며 색상만으로 순서를 전달하지 않는다.

### 5.2 상태와 상호작용

- 검색 전에는 `지정할 계정` select를 비활성화한다.
- 검색 요청 중에는 검색 버튼에 `검색 중…`을 표시하고 중복 요청을 막는다.
- 검색 결과가 없으면 1단계 안에 `검색된 계정이 없습니다.`를 표시한다.
- 검색 결과가 있으면 계정 select를 활성화한다.
- 계정을 선택하기 전에는 `담당자로 지정` 버튼을 비활성화한다.
- 지정 요청 중에는 `담당자로 지정 중…`을 표시하고 모달 입력을 유지한다.
- 지정 실패 시 검색어, 후보 목록, 선택 계정, 역할을 유지해 다시 시도할
  수 있게 한다.
- 지정 성공 시에만 모달을 닫고 최신 조직 정보를 다시 불러온다.
- 기본 `닫기` action은 숨기고 하단의 `취소`로 닫기 동작을 통일한다.
  Escape와 focus 복원 동작은 기존 `Dialog`가 계속 담당한다.

### 5.3 반응형

- 데스크톱에서 모달은 현재보다 여유 있는 폭을 사용한다.
- 검색 입력과 버튼은 한 행에 배치하되 검색 입력이 남는 폭을 차지한다.
- 계정과 역할은 데스크톱에서 2열이다.
- 좁은 화면에서는 검색 입력·버튼과 계정·역할을 모두 단일 열로 바꾼다.
- 하단 action은 데스크톱에서 오른쪽 정렬하고 모바일에서는 사용 가능한
  폭 안에서 줄바꿈한다.

## 6. 조직 요약 표현

### 6.1 값의 의미

세 화면에서 다음 표현을 사용한다.

| 레이블 | 값 |
| --- | --- |
| 대표 조직장 | 대표 이름 또는 `미지정` |
| 담당자 | 대표 조직장을 포함한 전체 담당자 수와 `명` |
| 프로젝트 | 연결 프로젝트 수와 `개` |

예를 들어 대표 조직장 1명과 추가 관리자 2명이 있으면 다음과 같다.

- 대표 조직장: `홍길동`
- 담당자: `3명`
- 프로젝트: `1개`

대표 조직장이 없고 추가 관리자도 없으면 `미지정`, `0명`으로 표시한다.

### 6.2 계산

현재 API의 `managerCount`는 추가 관리자 수다. 전체 담당자 수는 프런트
공통 함수에서 다음처럼 계산한다.

```text
managerCount + (primaryLeader가 있으면 1, 없으면 0)
```

조직 목록, 조직 상세, 프로젝트 조직 화면은 모두 이 함수를 사용한다.
계산을 각 컴포넌트에 반복 구현하지 않는다.

### 6.3 화면별 레이아웃

- 조직 목록 카드: `대표 조직장`, `담당자`, `프로젝트`를 옅은 배경의
  독립된 세 정보 칸으로 표시한다.
- 조직 상세 상단: 기존 메타 정보 배치를 유지하고 값을 `미지정`,
  `N명`, `N개`로 단순화한다.
- 프로젝트 조직 카드: 기존 카드 구조와 관리 링크를 유지하고 같은 값
  표현과 전체 담당자 계산을 사용한다.

## 7. 컴포넌트 경계

- `OrganizationManagersPanel`은 검색·지정 상태와 두 단계 모달을 담당한다.
- 공통 전체 담당자 계산 함수는 표시용 데이터 계산만 수행한다.
- `OrganizationsPage`, `OrganizationDetailPage`,
  `ProjectOrganizationsPanel`은 각 화면의 레이아웃을 소유하며 공통 계산
  함수만 사용한다.
- 서로 다른 화면 레이아웃을 하나의 거대한 요약 컴포넌트로 합치지 않는다.

## 8. 오류와 로딩 처리

- 계정 검색 오류는 1단계 안에서 표시하며 이전 검색어를 유지한다.
- 담당자 지정 오류는 2단계 또는 action 영역 가까이에 표시하며 검색
  결과와 선택값을 유지한다.
- 검색과 지정의 loading 상태는 서로 독립적이다.
- 서버 성공 전 조직 담당자 데이터나 요약 숫자를 낙관적으로 바꾸지 않는다.
- 지정 성공 후 최신 조직 조회가 실패하면 기존 성공 안내와 재조회 실패
  안내를 구분하는 현재 정책을 유지한다.

## 9. 접근성

- 두 단계 제목은 문서 구조상 식별 가능한 heading으로 제공한다.
- 번호 배지에는 텍스트 단계 제목이 함께 있어 색상에 의존하지 않는다.
- 비활성 select와 버튼은 native `disabled`를 사용한다.
- loading 버튼은 기존 `Button.loading`의 `aria-busy`와 진행 문구를 사용한다.
- 오류·검색 결과 없음 안내는 기존 status message 패턴을 사용한다.
- Dialog의 focus trap, Escape 닫기, 닫힌 뒤 focus 복원 동작을 유지한다.

## 10. 테스트

### 조직 담당자 지정 모달

- 두 단계 제목과 하단 `취소`, `담당자로 지정` 동작을 렌더링한다.
- 검색 전 계정 select와 지정 버튼이 비활성화된다.
- 검색 중 진행 문구와 중복 요청 차단을 확인한다.
- 검색 결과가 없을 때 `검색된 계정이 없습니다.`를 표시한다.
- 검색 성공 후 후보를 선택하고 역할을 지정할 수 있다.
- 지정 실패 후 검색어, 후보, 선택 계정, 역할이 유지된다.
- 지정 성공 후에만 모달이 닫힌다.
- 모바일 CSS에서 두 단계 입력이 단일 열로 전환된다.

### 조직 요약

- 대표 없음, 추가 관리자 0명: `미지정`, `0명`
- 대표 1명, 추가 관리자 0명: 대표 이름, `1명`
- 대표 1명, 추가 관리자 2명: 대표 이름, `3명`
- 위 의미를 조직 목록, 조직 상세, 프로젝트 조직 카드에서 동일하게
  검증한다.
- 프로젝트 수는 중복 문구 없이 `0개`, `1개`로 표시한다.

## 11. 배포 영향

프런트 정적 자산만 변경된다. API, D1 migration, Worker secret, Cron 설정은
변경하지 않는다. 기존 Cloudflare Worker 배포 절차의 테스트, type check,
Web build, dry-run을 통과한 뒤 동일 Worker에 재배포한다.
